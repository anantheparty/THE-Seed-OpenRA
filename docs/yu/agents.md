# yu Knowledge Base

- Deliverables explicitly assigned by Wang go under `docs/wang/` (or `docs/wang/archive/` for archived investigation artifacts); `docs/yu/` is reserved for yu's own agent state and yu-only files.
- When a subagent is already auditing a concrete bug chain, avoid doing the same read-only reconstruction locally in parallel; use local time for complementary hypotheses, code-path tracing, or state maintenance instead of duplicate log review.
- Base `TaskAgent` now has native `push_player_response()` intake that re-injects `PlayerResponse` as a normal wake event; WS / Adjutant reply flow can route through `Kernel.submit_player_response()` directly.
- Test suites that replace the shared runtime surface with custom mocks can hide integration blockers; before clearing a milestone, check the mock API shape against the real `openra_api` / `Kernel` / `WorldModel` contracts.
- `openra_api.GameAPI` is stateless per request, so OpenRA process restarts do not require recreating the API client; runtime reattachment only needs `wait_for_api()` plus `WorldModel.reset_snapshot()` and a forced refresh.
- In the live runtime, `GameLoop` currently runs on the same asyncio loop as WS/Adjutant/LLM coroutines; because `WorldModel.refresh()` and job ticks call synchronous `GameAPI` sockets, slow game polling can starve `asyncio.wait_for()` and make LLM calls appear to hang even when the provider itself is healthy.
- When `WorldModel.refresh_health()` reports `stale=true`, Adjutant should fail closed on stateful commands and query responses instead of letting LLM or direct routes answer from the last snapshot; use a clear degraded response and keep direct deploy/rule/NLU handlers guarded individually.
- In `OpenCodeAlert`, GameAPI request spam can come from `CopilotCommandServer.AcceptAsync()` connection logs rather than the JSON payload dumps; the payload dumps are already behind `CopilotDebug`, but per-connection accept/stop-loop logs must also be gated if the Python side opens one short-lived socket per request.
- `GameAPI` long-connection reuse requires protocol framing as well as socket reuse: the old `send once -> wait for EOF/timeout` behavior works only because the server closed the socket after each response. To support persistent connections safely, both client and `CopilotCommandServer` must use a real message boundary (newline framing here), with fallback parsing for one-shot legacy packets.
- Kernel actor allocation must treat `building` / `static` actors as opt-in resources. Soft needs like `{"owner": "self"}` should only claim mobile actors by default; jobs that truly want structures must say so explicitly in their `ResourceNeed` predicates.
- Ordinary task runtime facts must not surface capability-only buildability/feasibility hints; those should be injected only for capability tasks so managed tasks do not infer they can build prerequisites themselves.
- For emergency auto-responses such as `BASE_UNDER_ATTACK`, creating a managed task and waiting for TaskAgent/LLM is too slow for first reaction. The Kernel should directly start or retarget a concrete expert job, and let the task/LLM layer handle follow-up reasoning afterward.
- `BASE_UNDER_ATTACK` also needs a kernel-level player warning path. Broadcast + auto-response alone leaves the first reaction visually silent; the immediate “基地受到攻击，开始反击” notification must be emitted at runtime event routing, not deferred to task/LLM progress.
- User-initiated synchronous feedback (`command`, `reply`, `query`) should go through the chat/adjutant response channel, not `player_notification`. `player_notification` is suitable for async system/task events; routing direct command acks there makes the UI feel like the command was swallowed.
- For `task_question` clarification surfaces, a visible “取消任务” action must call the existing `command_cancel` path with `task_id`; sending `"取消任务"` as a normal `question_reply` answer only hands text back to the TaskAgent and does not guarantee real cancellation.
- In the Vue chat frontend, long-lived `localStorage` history plus top-level WS listener registration can create “stale cached dialogue” and duplicate message replay. Chat history should be session-scoped, and WS subscriptions should unregister on component unmount.
- In the Vite-served frontend, voice `fetch('/api/asr'|'/api/tts')` must not derive its base URL from `window.location.port` blindly. If the page is on the Vite dev server (`:5173`), that sends voice HTTP to Vite instead of the aiohttp backend and produces a non-JSON `404`, with no corresponding backend ASR log because the request never reaches `WSServer`.
- Even with the backend URL fixed, browser voice fetches from the Vite dev server are cross-origin (`5173` -> `8765`). `WSServer` must answer `OPTIONS /api/asr` and `OPTIONS /api/tts` with CORS headers; otherwise the browser surfaces a generic `Failed to fetch` before the ASR/TTS handlers run.
- Browser `MediaRecorder` audio arrives as `webm/opus` in the current frontend, but DashScope ASR does not accept that container directly. The backend ASR layer must transcode browser uploads such as `webm` / `ogg` / `opus` to `wav` before SDK invocation.
- For browser E2E stability, the preferred voice path is now frontend-side WAV generation before upload. Backend `ffmpeg` transcoding remains only as a fallback for unexpected/non-browser clients; relying on backend-only `webm` conversion still leaves room for malformed browser fragments to fail before ASR.
- Frontend-side WAV generation must be fail-open. Browser `decodeAudioData()` is not reliable on every `MediaRecorder` fragment; if frontend decoding/rendering fails, `prepareAsrUpload()` should fall back to the original blob so the backend can still attempt ASR/transcoding.
- `LOW_POWER` is a hybrid signal: capability may consume it for planning only when real demand exists, but the user-visible warning cannot depend on capability wake/LLM execution. If capability idle gating is strict, the low-power warning must be surfaced directly from kernel event orchestration or another unconditional notification path.
- In Adjutant command parsing, repair-facility nouns (`维修厂` / `修理厂` / `维修中心` / `修理中心`) must not be treated as repair verbs. Otherwise build commands get swallowed by the repair short-circuit and return `当前没有维修厂，无法执行回修` before economy/NLU routing can run.
- Repair commands must distinguish vehicle repair from building repair. `维修厂` is a prerequisite for vehicle repair feedback/routing, but damaged buildings should still route to `RepairExpert` without a repair-facility check; otherwise `修理电厂` is falsely blocked by `当前没有维修厂`.
- The `task_list` payload already carries enough per-job metadata (`status`, `summary`, `expert_type`) for the frontend to infer "waiting on capability/resource" hints without changing task_agent/kernel payloads.
- `logging_system` persists per-session `all.jsonl` / `components/*.jsonl` / `tasks/*.jsonl`, but the runtime replay path is still memory-only; `sync_request` replays current live state plus recent in-memory logs, not a loaded historical session.
- The current real aiohttp/runtime smoke covers live `sync_request`, `command_submit`, `question_reply`, `command_cancel`, `session_clear`, `game_restart`, requester-scoped `task_replay`, and unknown-message errors end to end through a real runtime.
- `main_module.main([...])` coverage is not enough by itself; keep the pytest-backed direct-entry smoke that exercises the true `if __name__ == "__main__": raise SystemExit(main())` path and the shutdown wrapper, so startup/runtime regressions do not hide behind helper-only coverage.
- If `Adjutant` feels too weak as top-level coordinator, the first check should be context plumbing, not a new planner layer: `main._build_task_triage()`, `WorldModel.compute_runtime_facts()`, and `Kernel._sync_world_runtime()` already produce coordinator-useful state (`triage`, `capability_status`, `unit_reservations`, `info_experts`) that `Adjutant._build_context()` and `_battlefield_snapshot()` still under-consume.
- `Kernel` syncs both `unfulfilled_requests` and `unit_reservations` through `runtime_state` / `compute_runtime_facts()`, and capability context renders both `[待处理请求]` and `[预留]`.
- `Kernel` maintains a task actor group registry for fulfilled request handoff; `active_tasks` runtime_state carries `active_actor_ids` / `active_group_size`, which exposes task-owned unit continuity without changing queue semantics.
- `request_units` semantics need two separate thresholds: `blocking=False` reinforcement requests must not suspend the agent at all, and blocking requests should resume once `min_start_package` is satisfied rather than waiting for full `count`. Treating any pending/partial request as "still blocking" breaks the whole start-package contract.
- `Adjutant` consumes `capability_status` in three separate places: `_coordinator_snapshot()` (top-level summary), `_derive_task_triage()` (active task status lines), and `_try_merge_to_capability()` (merge acknowledgement). Adding new capability lifecycle fields in only one or two of these paths leaves the coordinator and the task list disagreeing about capability state.
- Building production needs two extra rules that unit production doesn't: `EconomyJob` must call `GameAPI.produce(..., auto_place_building=True)` for `queue_type="Building"`, and it must treat `has_ready_item` / `done` queue entries as first-class queue state. Otherwise ready buildings sit unplaced, block the build queue, and the task layer doesn't realize the queue is jammed.
- Simple bootstrapped structure-build tasks need a deterministic close path. If the TaskAgent sends a one-shot `EconomyExpert(Building)` job and then re-enters the LLM after that job succeeds, the model can hijack the task into unrelated follow-up work.
- For `EconomyJob(queue_type="Building")`, `PRODUCTION_COMPLETE` is not enough to declare success. The task is only really complete after the final ready building has been placed and the queue state confirms the ready item is gone.
- Pre-existing matching ready buildings in the `Building` queue must count toward the requesting task's completion. Otherwise a task can legitimately place the desired building, clear the queue, and still hang forever waiting for a completion event that belongs to a past task.
- `SignalKind.BLOCKED` needs a user-visible path. Routing it only to the TaskAgent leaves live commands looking silently stuck; the Kernel should register a `TaskMessage(task_warning)` so the dashboard/adjutant can surface the real blocker reason.
- `Adjutant.handle_player_input()` is still fundamentally a precedence chain, not a neutral two-stage classifier. Ordering is behavioral truth: early feedback/rule/NLU gates can steal a sentence before capability merge or LLM classification sees it, so route changes must be reasoned about as precedence changes, not only intent changes.
- `RuntimeNLU` currently sits before economy capability merge. If a short economic command starts bypassing capability dedupe/merge semantics, the first suspect is direct NLU preemption rather than capability logic itself.
- The authoritative RA production registry lives in `OpenCodeAlert/mods/ra/rules/{structures,vehicles,infantry}.yaml`, and Wang also extracted the common IDs explicitly: buildings such as `POWR/APWR/PROC/WEAP/BARR/TENT`, vehicles such as `HARV/MCV/1TNK/2TNK/3TNK/V2RL`, and infantry such as `E1/E2/E3/E6`. When production-name translation gets ambiguous, prefer this ruleset-backed ID registry over freeform English labels.
- Roadmap docs must be split by maturity: current execution references, near-term bridge docs, and future target-state docs should not share the same backlog language. `README.md` / `PROJECT_STRUCTURE.md` should describe live runtime truth, while `design_v_next.md` / `expert_redesign.md` must be explicitly labeled as future-only.
- Hard unit ownership is already supported at the action surface: `Recon` / `Combat` / `Movement` accept explicit `actor_ids` through config, tool schema, handler pass-through, expert `get_resource_needs()`, and kernel fallback inference.
- Capability truth is currently split across three layers: `unit_registry.py` (raw YAML metadata), `openra_state/data/dataset.py` (normalized demo prerequisite/buildability table), and `experts/knowledge.py` / `task_agent/context.py` (expert and capability roster filters). Any capability-facing prerequisite/buildability change must be checked across all three layers to avoid truth drift.
- Demo capability roster text is sourced from `openra_state/data/dataset.py` via `_DEMO_{SHARED,FACTION}_CAPABILITY_ROSTER` plus `_DEMO_PROMPT_DISPLAY_NAME_OVERRIDES`, then injected into `task_agent/policy.py`. If prompt text omits or misnames a demo building/unit, fix the dataset roster source instead of patching prompt strings locally.
- `openra_api.production_names.production_name_entry()` and `production_name_unit_id()` are the shared registry-backed alias lookup path for unit/category/name resolution; `world_model`, `IntelService`, `StructureData`, and `CombatData` should delegate there instead of rebuilding ad hoc alias tables.
- `openra_state/data/dataset.py` currently overwrites faction-specific rows when the same unit id is registered twice (e.g. Allied/Soviet `E1/E3/E6`), and `WorldModel.compute_runtime_facts()` still hardcodes `faction="soviet"`. Any future faction-aware capability/buildability work must normalize faction truth explicitly instead of assuming the dataset map is lossless.
- In `Adjutant`, `_build_context()` is safest when it consumes one frozen build-snapshot object for both coordinator inputs and per-task kernel artifacts. Leaving `_coordinator_snapshot()` able to self-collect inputs makes non-atomic context regressions easy to reintroduce later.
- In Adjutant task query focus, task-local unit-pipeline counts (`remaining/assigned/produced/start_released`) should prefer `build_task_unit_pipeline_focus()` runtime truth over triage snapshot defaults. `TaskTriageSnapshot` normalizes missing integers to `0`, so `triage or runtime` silently hides real partial-progress values.
- Generic capability umbrella directives like `发展经济` / `继续发展科技` should merge and surface current capability state, not hit recent-directive dedupe. Concrete production directives (`电厂`, `优先补电`, counted build/produce text) are the dedupe-safe surface; broad strategy reminders are not.
- In test-governance slices around Adjutant, exact internal call-count assertions are usually the wrong contract. Route-precedence tests should pin ordered subsequences and forbidden branches, because harmless extra guard checks (`economy_check`, snapshot refresh, etc.) will otherwise cause noisy red tests without a product regression.
- Shared Adjutant test doubles now belong in `tests/_adjutant_fixtures.py`. Slice 0 scope is only the common baseline mocks (`MockTask`, `MockKernel`, `MockWorldModel`); probe subclasses and file-specific helpers should stay local.
- `tests/test_capability_task.py` is not a direct slice-1 consumer of `tests/_adjutant_fixtures.py`: its local Adjutant-routing `MockKernel` omits `jobs_for_task`, while current Adjutant capability/runtime-NLU paths require that method. Reusing the shared fixture there needs a thin local adapter or a separate mock cleanup, not blind replacement.
- For capability truth, disabled prerequisite providers must not keep unlocking downstream buildables. In OpenRA, a disabled `ProvidesPrerequisite` trait stops contributing tech-tree truth, so runtime `buildable` should filter against active prerequisite providers, while `production_readiness_for()` should distinguish `missing_prerequisite`, `producer_disabled`, and `disabled_prerequisite`.
- Planner faction inference must not rely on demo-roster-only canonicalization. Using `demo_capability_unit_type_for()` on own actors drops faction-specific non-demo ids like `tent` / `2tnk`, then ambiguous localized names such as `兵营` can collapse to `barr` and misclassify an Allied base as Soviet.
- Adjutant deploy short-circuit should use runtime facts as the truth source for `has_construction_yard` / `mcv_count`, but only treat the deploy state as ambiguous when runtime facts claim an undeployed MCV exists and `query("my_actors", {"category": "mcv"})` cannot return a concrete actor id. A construction-yard mismatch alone is not a safe blocker for `rule_deploy_missing_mcv` / `rule_deploy_already_deployed`.
- In `Adjutant.handle_player_input()`, runtime NLU must run before the generic economy-capability merge. If `_is_economy_command()` wins first, short build nouns and composite produce phrases collapse into one raw Capability directive and lose NLU decomposition (`兵营` stops being a direct produce route; `电厂兵营，然后5步兵` degrades to one wake/one step).
- Recon/combat follow-up routing must get a chance before direct runtime-NLU/rule execution. Otherwise utterances like `继续探索左下角` or `全部步兵去探索地图` are consumed as fresh direct routes and open duplicate recon tasks instead of merging into the active task.
- Old-NLU implicit-produce heuristics must reject counterattack / interception language (`反击` / `回击` / `拦截` / `守住` etc.). Otherwise short phrases such as `防空反击一下` can be coerced into `produce` by both `CommandRouter` and runtime-NLU safe overrides, surfacing as mistaken `EconomyExpert` / `[NLU直达]` actions.
- Runtime NLU `produce` fallback for glued production nouns must dedupe by matched text span/alias, not by raw unit id. Otherwise shared aliases like `兵营` can double-expand into both `BARR` and `TENT`.
- TaskAgent prompt roster text must be faction-scoped in the live demo policy surface (`soviet` today), not the union roster. Otherwise ordinary tasks and `EconomyCapability` both see Allied-only units like `jeep` / `1tnk` / `2tnk` and plan against impossible current-state options.
- Demo defense structures (`tsla` / `ftur` / `sam`) belong to the `Defense` production queue even when the capability roster advertises them alongside buildable structures. Capability/demo truth must keep `queue_type=Defense`, or queue filtering/blocker checks drift from real OpenRA production state.
- `EconomyCapability`'s fail-closed demand gate and prompt can both suppress legitimate follow-through after a player economy directive. The current minimal fix is a sticky `active_directive` derived from recent capability player inputs: ignore internal bracket-prefixed notes (`[Kernel fast-path]`, `[NLU直达]`, etc.), keep only the latest user-style directive, and expose it through runtime projection/context/prompt so capability may continue one smallest step even after `[玩家追加指令]` returns to `无`.
- Capability-side `[基地推进]` / `base_progression` is planning truth, not execution authorization. In idle startup logs, if the capability is allowed to reach the LLM with `[玩家追加指令] 无` and no requests, the model can still treat `基地车待展开 -> 电厂 -> 矿场 -> 兵营 -> 车厂` as an implicit bootstrap script and autonomously dispatch `DeployExpert` / `EconomyExpert` jobs.
- Concrete player economy commands that already resolve to an `EconomyExpert` action (`电厂`, `兵营`, `步兵3`, concrete build/produce rules, and NLU decomposed production steps) must not split ownership between a standalone task and `EconomyCapability`. If capability exists, the preferred path is capability-owned execution plus non-waking `NLU直达` history; standalone direct tasks are only a fallback when capability is absent or the command shape is not yet supported by the capability fast-path.
- Single-step concrete economy NLU/rule commands must stay under `EconomyCapability` ownership when capability exists. The wrong pattern is “open a standalone managed task, then also notify capability”; that produces duplicate builds. The correct pattern is “start the `EconomyExpert` job under capability, and record a non-waking `[NLU直达] ...` note/history entry”.
- `Adjutant.handle_player_input()` runs `_maybe_handle_attack_feedback()` before runtime NLU, coordinator-hint merge, and LLM classification. Any utterance containing an attack keyword plus a unit/building token can be rejected as `rule_attack_missing_target` if no visible enemy actor matches that token, even when the player is actually giving a composite preparation command (for example `整一大批步兵和防空车，准备一轮进攻`).
- `openra_state.data.dataset._HINT_TO_UNIT` is authoritative for task `request_units(...)` hint inference before bootstrap. If Soviet demo hints like `ftrk` / `防空车` / `mig` / `yak` are missing there, `infer_unit_type_for_request()` falls back to category defaults (`vehicle -> 3tnk`, `aircraft -> None`) and the whole request/bootstrap chain drifts to the wrong unit type.
- A browser ASR traceback that reaches `voice/asr.py::_normalize_audio_input()` with `audio_format='webm'` proves the backend code is running; it specifically means the preferred frontend WAV conversion failed and `prepareAsrUpload()` fell back to uploading the original `webm`.
- In the chat frontend, ASR success must reuse the same submit path as typed input. Writing recognized text back into `inputText` alone silently regresses voice "auto send" even when recognition itself succeeds.
- Runtime NLU `produce` cannot rely on confidence alone for glued multi-target phrases. The old router can assign one trailing quantity to the first matched production alias (for example `建造电厂兵营五个步兵` -> `powr x5`) with near-1.0 confidence, so ambiguous multi-target/no-connector produce text must fail closed to Capability/Agent instead of direct execution.
- Runtime NLU multi-item fallback must ignore overlapping alias spans. Otherwise one concrete noun can be split into a long valid target and a shorter generic alias at the same position (for example `防空车` -> `ftrk` + `sam` via `防空`), which incorrectly turns a valid single-target command like `五个防空车` into a fake multi-target ambiguity and blocks NLU execution.
- `_maybe_route_active_task_followup(...)` runs before RuntimeNLU, so it must explicitly fail closed on economy/build text. Single-character combat hints like `防` are too coarse: commands such as `建造防空塔` can otherwise be misclassified as combat follow-ups and get injected into active battle tasks before the economy path sees them.
- `Adjutant._match_build()` must stay narrower than general economy detection: it is only safe for single-target `count=1` build short commands. Counted or multi-target build text (for example `建造两个矿场一个车间`) must fail closed to Capability, or the rule path will silently drop quantity and start whichever structure alias matched first.
- Plain retreat shouts (`撤退撤退撤退`, `全军撤退...全部回来...`) must route through the direct retreat rule instead of becoming generic managed tasks. If they fall through to TaskAgent/LLM, the agent can wrongly `request_units()` and create overlapping retreat ownership instead of simply pulling existing units back.
- Direct retreat routing is not enough by itself: before starting a new retreat `MovementExpert`, Adjutant must preempt conflicting active combat/recon tasks, or the old tasks keep their claims and the operator sees contradictory “继续攻击还是等待撤退” prompts over the same force package.
- Capability serialism comes from prompt policy more than handler hard-limits. `produce_units` only blocks duplicate same-item submissions; the stronger “一次一个最小里程碑 / 每次 wake 一步” behavior is enforced by `task_agent/policy.py`.
- Pending-question reply routing has two layers: multi-reply was already early-routed, but single short replies were previously delayed until later classification. That let words like `需要` get swallowed by continuation/NLU/new-command paths. Single pending-question short replies must be routed before continuation/runtime NLU.
- Runtime NLU composite safety must validate each `produce` clause independently. The old parser can map `基地` to `建造厂(fact)`, so goal phrases like `找到敌方基地` may be misread as `produce fact`; without a per-clause safety gate, recon commands like `探索地图，找到敌方基地` get wrongly split into `Recon + Economy`.
- Adjutant duplicate-task overlap must weight action domain over generic target nouns. Shared object words like `敌方` / `基地` are enough to dedupe same-domain recon goal variants, but they must not collapse recon discovery and combat harassment/attack into one task.
- Attack feedback must not use generic enemy-base phrases as explicit visible-target requirements. Text like `进攻敌方基地` / `基地残留位置` should route to known enemy positions (including frozen fog positions) instead of being blocked as missing `建造厂`/`火箭兵` targets.
- `撤退回基地` is not a planning task. Route it as a bounded direct `MovementExpert` retreat using task-owned active actor ids when available, and target a base-adjacent offset instead of the construction-yard tile itself; otherwise it drifts into a managed TaskAgent loop that can request fresh units after movement failure.
- `Movement/Combat/Recon` now support partial explicit-group startup via `wait_for_full_group=False` plus `min_ready_count` (0 = auto, currently capped to a 3-unit starter package). This is implemented as an aggregate `actor_ids_any` resource need layered on top of exact per-actor needs, so late stragglers can still join after the job has already started.
- `MovementExpert` also supports partial completion via `min_complete_count` (0 = auto). When `wait_for_full_group=False`, retreat/move can succeed once a sufficient subset has reached the destination instead of waiting for every surviving actor in the original explicit group.
- In the Capability prompt, “显式多目标清单” must be stated as the current minimal closure, not a loose exception. If that hierarchy is implicit, the model over-weights “一次一个最小里程碑” and serializes commands like `电厂、兵营、5个步兵` into multiple wakes even when all items are already buildable now.
- `capability_status.phase="idle"` must mean “no requests, no jobs, and no active directive.” If `active_directive` is still present, the exported phase should move to a distinct non-idle value (now `directive_pending`) so UI, Adjutant, and Capability prompt do not see contradictory truth.
- Runtime NLU multi-produce direct execution must be gated by live `buildable_now`, not only text parsing. If a pure economy command names targets that are not all immediately buildable now, fail closed to Capability merge instead of starting blocked direct `EconomyExpert` jobs.
- Ordinary combat/recon tasks already receive usable force-allocation context: `runtime_facts` includes `active_actor_ids` / `active_group_size`, `world_summary.military` includes `idle_self_units`, and `Adjutant` computes `free_combat_units`. The main drift is not “agent看不见兵力”, but that these signals are underused or bypassed by bad routing/workflow.
- `CombatExpert.get_resource_needs()` currently treats `actor_ids=None` and `unit_count<=0` as `count=999` ("all available"). Any task/job that starts `CombatExpert` without explicit actors or a bounded unit count effectively asks Kernel for the whole global combat pool, which amplifies parallel-task conflicts and produces misleading `resource_lost` counts like “Missing 963 actor resource(s)”.
- Operator-wide commands such as `全军出击` or `现有单位也移动过去` must not be expressed through ordinary task-local ownership semantics. Adjutant should resolve the live force package directly, preempt conflicting non-capability combat/recon/movement tasks, and start the replacement expert with explicit `actor_ids`.
- Idle startup bootstrap is not a valid Capability authorization source. `base_progression` / `[基地推进]` may describe the next recovery step, but with no player message, no active directive, and no unfulfilled request, EconomyCapability must idle-skip instead of reaching the LLM.
- Capability `active_directive` is not uniformly long-lived. Short alarm-style defense phrases such as `家里被打了` / `基地被打了` must expire quickly instead of staying sticky for minutes; otherwise EconomyCapability keeps treating them as open-ended production goals after the immediate threat has passed.
- Adjutant `rule_recon` is a separate direct path from runtime NLU and must preserve requested scout count. If it ignores quantities, commands like `五个步兵探索一下地图，找到敌方基地` silently start `ReconExpert(scout_count=1)` before TaskAgent/LLM ever runs.
- `request_units(...)` idle fulfillment currently matches by category first and uses `hint_match_score(...)` only for sorting, not for admission. That means a request like `category=vehicle, hint=重坦` can legally absorb an idle `V2` when no exact heavy tank is idle, producing mixed task actor groups before bootstrap production finishes.
- Adjutant query answers are still built from coarse `world_summary + battlefield_snapshot + active_tasks` rather than exact per-task runtime/job truth. For task-specific questions like `为什么只有3个上了`, this can degrade into confident but wrong narrative even when the task already has exact `active_actor_ids`, child jobs, and `resource_lost` evidence.
- `_maybe_handle_attack_feedback(...)` must fail closed on build/use-unit phrases with generic enemy targets. Inputs like `建造五个火箭兵去攻击敌方目标` or `用火箭兵攻击敌方目标` are not “missing visible 火箭兵 target” errors; they are economy/combat routing cases and should bypass the explicit-target short-circuit.
- Generic command disposition routing must never treat `EconomyCapability` as a replaceable ordinary task. Capability-owned merge semantics go through `_try_merge_to_capability(...)`; if classification/coordinator hints target the capability task with `override` / generic `merge`, the common `_handle_command_with_disposition(...)` path must fail closed instead of cancelling or injecting via ordinary task-followup logic.
- Runtime-NLU operator-wide attack is a separate boundary from direct rule attack. Even when `全军出击` / `全员出击` slips onto the NLU path, `Adjutant` must still resolve explicit live `actor_ids`, preserve them through `_resolve_attack_step(...)`, and preempt conflicting non-capability combat/recon/movement tasks before starting the replacement combat job.
- Request-linked production only becomes truthful when the kernel itself credits the produced actor back into `UnitRequest` / `UnitReservation`; carrying `request_id` / `reservation_id` inside `EconomyExpert` signals is not enough unless `Kernel.route_signal()` consumes that ownership metadata.
- For request-linked production, `PRODUCTION_COMPLETE` / `UNIT_IDLE` must run request fulfillment before generic rebalance, and `WorldModel` must emit `UNIT_IDLE` for newly appeared self-idle units as well as busy→idle transitions; otherwise fresh actors can be claimed by generic jobs before waiting requests ever see them.
- For shorthand multi-target economy text that runtime NLU rejects as ambiguous (for example `兵营3步兵`), the stable fail-closed behavior is Capability merge, not ordinary managed-task creation and not relaxed direct-job inference. Preserve the ambiguity guard; widen the economy fallback instead of inventing count ownership.
- Direct rule families must reject query-shaped phrasing inside the matcher itself, not only in outer feedback helpers. If the matcher still accepts the text, `handle_player_input()` can trigger the rule before query/classification logic sees it.
- Ultra-short combat shouts like `你打。` / `上。` should never fall through to generic managed-task creation. Front door must either merge them into one active combat task or stop and ask for a concrete target/mode.
- Operator pullback corrections like `拉回来` / `都回来` / `别去那边` belong on the existing retreat/preemption path. They are runtime corrections, not fresh planning tasks.
- `override` / `interrupt` disposition handling must not bypass deterministic direct routing. If disposition is applied before combat/retreat/operator-wide execution, the handler must try the same bounded rule/NLU execution first and only then fall back to generic task creation.
- Attack-preparation language must be guarded in the real attack lanes, not only in `_maybe_handle_attack_feedback()`. If `准备进攻...` is only excluded from negative-feedback short-circuits, `_match_attack()` or runtime NLU can still turn it into immediate combat execution.
- Query/task explanation surfaces must carry raw per-task runtime truth separately from triage-derived summaries. Otherwise fields like `active_expert` / `blocking_reason` collapse into coarse `task_warning` style narratives exactly where the operator expects exact task-owned causality.
- Mixed economy+combat utterances (`造兵后进攻`, `建造X去攻击Y`) must not fall into direct `EconomyExpert` or capability-merge lanes just because the NLU/router sees `produce`. In the current architecture they should become managed workflow tasks so the `produce_units_then_attack` contract stays truthful end to end.
- Glued shorthand economy composites can escape both safe direct NLU and capability merge: `RuntimeNLU._steps_from_intent("produce")` intentionally drops ambiguous multi-target counted text like `兵营3步兵`, but unless `Adjutant._is_economy_command()` also recognizes the shorthand, the input falls through to ordinary managed-task creation instead of `EconomyCapability`.
- Unit-request hint matching must treat `hint` as an admission gate, not only a sort preference. Otherwise category-wide idle fulfillment (`vehicle`) will silently absorb unrelated units (for example V2s into a `重坦` request) and corrupt the task-owned force package before bootstrap production can finish.
- Operator-wide attack aliases such as `全面攻击`, `总攻`, and `Ready，全体发起进攻` must normalize to the explicit all-force attack lane. If they fall back to generic `nlu_attack` / managed-task handling, they lose live `actor_ids` resolution and stop behaving like true operator override commands.
- Same-domain combat/recon auto-merge must require either an explicit follow-up signal (`继续`, `然后`, `改`, `撤`, etc.) or clear text overlap with the existing task. “Only one active recon/combat task exists” is not enough by itself, or fresh targets like `去右上角侦察` get silently injected into old tasks.
- Runtime NLU safe-router override is too permissive for high-risk combat/recon unless intent alignment is enforced. For `attack` / `explore`, `pred.intent != route_intent` must fail closed; otherwise an `explore` prediction can still direct-execute `CombatExpert` just because the router text “looks like attack”.
- Capability directive dedupe must use canonical economy intent keys and must apply at both front-door entry points: generic `capability_merge` and direct `EconomyExpert` fast-paths. Otherwise equivalent short commands such as `电厂` / `建造电厂` bypass each other depending on whether runtime NLU or the economy merge wins first.
- Ordinary managed-task completion context must not serialize `_recent_reports` from other tasks. Cross-task reports are fine for debug surfaces, but when they enter the normal `TaskAgent` prompt they become borrowed completion evidence and push `partial`/`succeeded` summaries toward false ownership claims.
- Ordinary managed tasks must not use `query_world(threat_assessment)` as a completion-evidence shortcut. That query is just coarse `world_summary` in disguise; if it remains available, completion truth can still be laundered back into the prompt even after removing `[其他任务报告]`.
- For ordinary managed combat/recon tasks before they own units, prompt-only “request_units / wait / clarify” guidance is not enough by itself. If the prompt still exposes global `idle_self_units`, hides per-task request/reservation truth, or advertises generic combat fallback in tool schema, the LLM can drift back into global-idle narration or vague attack continuation even when handlers already reject unowned `attack`.
- Ordinary managed-task `unfulfilled_requests` / `unit_reservations` surfaces must be scoped by the current `task_id` before they feed workflow phase or `[单位调度]`. Kernel runtime projection is intentionally global for capability, but ordinary task context becomes misleading if it inherits other tasks' pending requests.
- Fail-closed owned-force guards for ordinary combat/recon tasks cannot depend on a one-off narrow raw-text regex. If the policy says the rule applies broadly, the matcher must include harass/raid variants such as `骚扰`, not only canonical `进攻/攻击/侦察` tokens.
- Ordinary managed-task context should not render global `idle_self_units` as a strong prompt hint. Even if handlers block `query_world(my_actors)`, surfacing `[世界] 我军X(闲置Y)` still nudges the LLM to treat global idle troops as implicitly available.
- Static tool descriptions are part of the runtime contract. If handlers now require task-owned units or `request_units` first, the corresponding `TOOL_DEFINITIONS` text must stop claiming Kernel will auto-supply a combat package, or the prompt drifts even when execution is already fail-closed.
- Prompt-governance tests should pin semantic invariants, not a single literal sentence fragment. Exact-string assertions on large prompts are a recurring false-red source when the rule still exists but the wording evolves.
- Ordinary managed tasks still need their own request/reservation truth, just not the full capability planning surface. The safe shape is a compact human-readable dispatch block, while raw `unfulfilled_requests` / `unit_reservations` stay hidden from the ordinary JSON header.
- Generic force-buildup attack phrasing such as `我需要更多的载具来进攻` must not depend on `_is_economy_command()` or concrete unit aliases to enter the bounded `produce_units_then_attack` lane; front-door mixed-workflow routing and ordinary workflow classification both need generic force nouns like `载具` / `部队` / `兵力`.
- Retreat conflict detection must not depend on `active_group_size > 0` for existing `MovementExpert` tasks. Direct retreat commands can legitimately create overlapping movement tasks while the runtime projection is temporarily zero-sized; treating those as “non-conflicting” causes repeated retreat orders to race for the same actor package.
- Prompt-contract tests should assert semantic clauses separately instead of one brittle full-sentence literal. Otherwise harmless wording/spacing changes keep turning owner suites red with no product regression.
- Explicit-group `actor_ids_any + min_ready_count` needs are startup gates, not advisory hints. Kernel resource assignment must keep the job `WAITING` until that aggregate package is satisfied; otherwise operator-wide commands begin running with an undersized force package after the first granted actor.
- Explicit-group late pickup also needs a rebalance edge when selected actors become idle. Without a dedicated `UNIT_IDLE`-style trigger, those jobs only recover on unrelated lifecycle / production / death events.
- For direct operator-wide jobs that target explicit `actor_ids`, `kernel.resource_assignment.find_unbound_resource()` currently only considers `idle_only=True` actors. After preempting/cancelling a combat job, the released actors can be unbound but still non-idle in the world, so a follow-up explicit-group movement/retreat job may stay `WAITING` with only an early `resource_lost` until a later `UNIT_IDLE` rebalance arrives.
- `cancel_task()` must mark the task terminal before aborting its jobs. Otherwise direct-managed job aborts emit `TASK_COMPLETE(aborted)` while the task still looks live, and `route_signal()` can transiently convert the same cancellation into `complete_task(...failed...)` before `task_cancelled(aborted)` lands.
- The safe fix for post-preemption actor reacquisition is narrower than “explicit ids ignore idle everywhere”: only `MovementExpert` explicit actor needs should be allowed to rebind `busy + unbound` actors. Leaving `CombatExpert` explicit groups on the old `idle_only` path preserves the existing startup-gate contract and avoids silently widening attack claims.
- Attack/recon lexicon drift across `Adjutant`, ordinary workflow classification, and `TaskAgent` owned-force guards creates real front-door truth leaks: if a combat verb is recognized in one layer but not the others, the same utterance can bypass direct routing, miss workflow gating, and regain forbidden `query_world(my_actors)` access.
- `MovementExpert` explicit-group completion truth must be evaluated against `config.actor_ids`, not the transient currently bound `self.resources` subset. Otherwise retreat/move tasks can lose bindings after `resource_lost`, move only the reacquired subset, and still report full success because `arrived_count` and auto `min_complete_count` shrink with the bound set.
- Ordinary managed commands that clearly mean “先生产/建造，再进攻” must enter `produce_units_then_attack` even when the raw text lacks an explicit unit noun. If workflow classification only keys on `attack + unit-token`, phrases like `先建造再攻击` fall back to free-form TaskAgent reasoning and start improvising build/tech choices instead of staying in the bounded `request_units / wait` lane.
- Shared task triage intentionally chooses one dominant state/status line, so request/reservation progress can disappear whenever a stronger local signal such as `task_warning` wins first. Task-specific query surfaces that need both “current warning” and “what units are still pending” must read task-local pipeline truth directly instead of assuming triage always carries `reservation_preview`.
- For unit-pipeline waits, structured triage fields alone are not enough: player-facing live task cards and cold replay both read `status_line` heavily, so compact progress (`assigned/produced/status/start`) must be formatted through a shared helper instead of letting replay and live surfaces invent separate generic summaries.
- Attack-family lexical contracts must stay aligned across front-door routing, ordinary workflow classification, and owned-force guards. If `Adjutant` learns a synonym like `袭击` but `TaskAgent` workflow/guard regexes do not, the same player intent splits across direct all-force routing, managed-task fallback, and global-force leakage.
- Direct explicit-group `MovementExpert` `resource_lost` / `progress` / `risk_alert` signals are currently log-visible but mostly not player-visible: `kernel.signal_delivery.route_expert_signal()` auto-registers task messages only for `BLOCKED`, while `task_triage` live inputs read `latest_warning` / `latest_info` only from registered task messages. Existing `group=active/requested` suffix helps a little, but it does not carry task-local missing/reacquired/arrived truth for explicit actor packages.
- Direct operator-wide retreat currently uses `wait_for_full_group=False` with no explicit `min_complete_count`, so `MovementExpert` falls back to `_AUTO_PARTIAL_GROUP_COUNT=3` and can mark `所有部队返回基地。` succeeded at `3/N` arrived even when the command semantics are full-force withdrawal.
